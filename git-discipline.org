#+begin_src python
from flask import Flask, request
from dotenv import load_dotenv
import requests
import re
import os

load_dotenv()
 
app = Flask(__name__)

@app.route('/')
def health_check():
    return '.'

@app.route('/linear', methods=['POST'])
def sync_linear():
    event = request.headers.get("x-event-key")
    payload = request.get_json()
    pr_data = payload.get("pullrequest", {})
    pr_title = pr_data.get("title", "")
    pr_url = pr_data.get("links", {}).get("html", {}).get("href")
    
    match = re.search(r'[A-Z]+-\d+', pr_title)
    if not match:
        return { "message": "No linear issue ID found"}, 202

    linear_issue_id = match.group(0)
    api_key = os.environ.get('LINEAR_API_KEY', '')

    mutation = """
    mutation CreateAttachment($issueId: String!, $url: String!, $title: String!) {
         attachmentCreate(input: {
           issueId: $issueId,
           url: $url,
           title: $title,
           iconUrl: "https://bitbucket.org/favicon.ico"
         }) {
           success
         }
       }
    """
    vars = {
            "issueId": linear_issue_id,
            "url": pr_url,
            "title": f"Bitbucket PR: {pr_title}"
        }

    
    r = requests.post(
        "https://api.linear.app/graphql",
        headers={"Authorization": api_key},
        json={"query": mutation, "variables": vars}
    )
    if r.status_code != 200:
        response = r.json()
        errors = response.get('errors')
        error = errors[0]
        content = { "message": error.get('message', '') }
        return content, r.status_code
    return { "message": "Success" }, 200

if __name__ == "__main__":
    app.run(debug=True, port=8000)

#+end_src

> What is trailing white spaces ?
> Git trailing whitespace refers to any spaces or tabs that appear after the last non-whitespace character on a line in a file tracked by Git. It is considered bad because it can lead to unnecessary changes in diffs, cause issues with line wrapping, and increase file size without adding any value to the code.

You can check if there any white space by running ~git diff --check~. The ideal solutions is to setup some kidn git pre commit hook to check if there are white spaces.

**** Git Stash
Looks like this will break our pipeline if we commit now. What we can do in this situation is /stash/ it in to the stack.
Use ~git stash push~ to push all the changes (staged and unstaged) to the stack.

There couple of options that is useful
> -m "<identifier>" - To give a name to the entry.
> --keep-index - To stash only the unstaged changes.
> --staged - To stash on the staged changes

Once you stash the changes you can see it by running ~git stash list~ and you can see the actual code by running ~git stash show -p {identifier}~

**** Merge Conflict
Ahh! OMG!!

Now we have a merge conflict. This is happening because we edited the same line twice with different branch. So git is having a hard time to decide which line to take.
This might seem very daunting. But actually it is far simpler than you can imagine. Since we made a change to the /pr-linking/ branch after creating the /pr-linking-tests/
this merge is not going to be a fast-forward merge. Because git needs to handle the commit we made on the source branch.

The log says in which file the conflict is. You can also check if the conflict is fixed by running ~git status~.
So if you open /main.py/ you will see the following.

#+begin_src python

@app.route('/linear-sync', methods=['POST'])
||||||| 06ff10c
@app.route('/linear', methods=['POST'])
=======
@app.route('/sync-linear', methods=['POST'])
>>>>>>> pr-linking-tests
#+end_src


So what is this ?
We have three version of a line we changed here. The top one is the change in the current HEAD ie. /pr-linking/ branch and the bottom one is the change coming from
merge source branch. ie /pr-linking-tests/.
And the middle one is the base for both. Basically means that was the common point from where all this changes happened in each branch.

So to fix this. Most of the code editors will have support. You can easily select one of the change or both.

[demo]

Or you can simply remove the ~<<<<<<HEAD ..........  >>>>>>pr-linking-tests~ to write the change you want.
After you can add the file to stage and continue with merge by ~git commit~ or ~git commit --continue~. Where you will write the merge commit message and complete the merge.

Now what about *git pull --rebase* ?
Well this is a common best practise you can do when you try to pull changes from remote. So imagine that you were working on a testing branch and you made some
commits. Now you want to push those changes to origin. But some updated the /origin/testing/ with some update. If you simply do ~git pull~ you are going to do a merge commit.
It is because you local contains some changes that is not there in the source.
So if you do ~git pull --rebase~ instead it will fetch all the remote changes and run all your local changes on top of that. Which will avoid having a merge commit.
One important thing to note is that similar to merge conflict you can get conflicts here as well. It is very less compared to merge. But you can do the same procedure to
fix the conflict. Once you are done with the fixing you can add the changes and continue rebase by ~git rebase --continue~.
